UAV平台重构实施计划

基于WRF气象驱动的无人机VRP智能路径规划系统 — 从旧架构迁移至 uav-platform-v2 新平台

状态: Phase 1 进行中 版本: v2.0.0-MVP 日期: 2026-06-12 模块: 12个Maven模块 算法: 93个组件

1 项目概述

本项目旨在将原有的无人机路径规划系统从旧架构全面重构为新一代微服务架构平台 uav-platform-v2。新平台基于WRF（Weather Research and Forecasting）气象数据驱动，集成多种先进的数据同化算法、AI预测模型和路径规划算法，实现多无人机协同任务规划与实时动态调整。

项目背景

原系统采用单体架构，随着算法复杂度增加（93个算法组件）和业务需求扩展，面临部署困难、扩展性受限、技术债务累积等问题。新平台采用领域驱动设计（DDD）思想，按业务域拆分服务，实现规划、同化、观测、适航、UTM五大核心领域的独立演进。

2.2 架构原则

多租户策略 架构

采用独立Schema隔离方案，每个租户拥有独立的数据库Schema。相比共享Schema+租户字段的方案，独立Schema在数据隔离性、性能优化和合规审计方面更具优势。

  • 动态数据源路由（Dynamic DataSource Routing）
  • Schema级连接池隔离
  • 租户上下文传递（TenantContext）
API版本控制 设计

采用Header-based版本控制策略，通过 Accept: application/vnd.uav.v1+json 请求头指定API版本。

  • 支持多版本并行运行
  • 版本协商与降级策略
  • 文档自动生成（OpenAPI 3.1）

2.3 分支与发布策略

Trunk Based Development 已确认

分支模型

  • 主干分支：main（长期稳定）
  • 特性分支：feature/*（短期存在）
  • 发布分支：release/*（按需创建）
  • 热修分支：hotfix/*（紧急修复）

质量门禁

  • 代码审查：最少 2人Review
  • CI检查：编译、单元测试、代码质量
  • 合并策略：Squash Merge
  • 发布策略：按需发布（非固定周期）

2.4 部署策略

滚动发布（Rolling Deployment）

采用滚动发布策略实现零停机部署。配合Kubernetes的RollingUpdate机制，逐步替换旧版本Pod，确保服务连续性。发布过程中实时监控关键指标（错误率、延迟、吞吐量），自动回滚异常版本。

迁移方法 方法
  • 直接迁移：Java算法直接重构
  • JNI桥接：C++算法通过JNI调用
  • Python集成：Py4J / GraalVM Python
  • 容器化：独立算法镜像（gRPC）

7.2 边云协同工作流

联邦学习流程 FederatedLearning
  1. 全局模型下发：云端下发初始模型至各边缘节点
  2. 本地训练：边缘设备使用本地数据训练模型
  3. 梯度聚合：边缘节点上传梯度（非原始数据）
  4. 全局更新：云端聚合梯度，更新全局模型
  5. 模型评估：验证全局模型性能
  6. 迭代循环：重复步骤1-5直至收敛

13.2 滚动发布流程

sequenceDiagram participant Dev as 开发者 participant Git as GitHub participant CI as CI Pipeline participant Registry as 镜像仓库 participant K8s as Kubernetes participant Monitor as 监控 Dev->>Git: Push to main Git->>CI: Trigger workflow CI->>CI: Compile & Test CI->>CI: Build Docker Image CI->>Registry: Push Image CI->>K8s: Deploy to Test K8s->>Monitor: Health Check Monitor-->>CI: Test Pass CI->>K8s: Deploy to Prod (Rolling) K8s->>K8s: Update 20% Pods K8s->>Monitor: Metrics Check Monitor-->>K8s: OK K8s->>K8s: Update 50% Pods K8s->>Monitor: Metrics Check Monitor-->>K8s: OK K8s->>K8s: Update 100% Pods K8s-->>CI: Deploy Complete

13.3 回滚策略

自动回滚触发条件
  • 错误率 > 1%（持续2分钟）
  • P99延迟 > 2秒（持续3分钟）
  • Pod重启次数 > 5次/10分钟
  • 健康检查失败 > 3次连续

触发后自动回滚至上一稳定版本，并发送告警通知。

14.3 代码审查规范

Trunk Based + 2人Review 已确认
  • 强制要求：所有合并到main的PR必须经过至少2人Approve
  • 审查内容：代码逻辑、测试覆盖、性能影响、安全合规
  • 自动化检查：SonarQube质量门禁、单元测试通过率、代码格式化
  • 审查时限：PR创建后24小时内完成首次Review
  • 冲突解决：作者负责Rebase，审查者验证
